# 向量数据库选型对比报告

> 为 VolSeek-Agent 项目选择持久化向量数据库的技术决策文档
> 当前状态：内存 VectorStore（暴力搜索，无持久化）→ 需替换为专业向量数据库

---

## 一、候选方案概览

对比范围：Qdrant、Milvus、Weaviate、Chroma、pgvector

## 二、核心对比表

| 维度 | **🥇 Qdrant** | Milvus | Weaviate | Chroma | pgvector |
|------|-------------|--------|----------|--------|----------|
| **Docker 部署** | ✅ **单容器** `qdrant/qdrant` | ❌ 多容器 etcd+minio+milvus | ✅ 单容器 | ✅ 单容器 | ✅ 单容器 |
| **Go 客户端** | ✅ `go-client/qdrant` 官方维护 | ⚠️ 旧 SDK 已归档，新 SDK 嵌入主库不稳定 | ⚠️ 仅 61 stars | ❌ **无 Go 客户端** | ✅ `pgvector/pgvector-go` 345 stars |

## 三、关键排除理由

### ❌ Chroma —— 直接排除
Chroma 没有官方 Go 客户端，只支持 Python / JS / Rust。Go 项目无法接入。

### ❌ Milvus —— 不推荐
- 即使 standalone 模式也需要 etcd + MinIO 两个额外容器，部署复杂度高
- Go SDK 旧版已归档（2025年3月），新版嵌入主库版本不稳定
- 对于中小规模场景（<10万向量）过于臃肿

### ⚠️ Weaviate —— 不推荐
- Go 客户端仅 61 stars，社区活跃度远低于服务端（16.6K stars）
- 容器镜像约 1GB，资源占用偏高
- Go 客户端代码生成质量不如 Qdrant 的 gRPC 自动生成客户端

### ⚠️ pgvector —— 可考虑但需额外部署 PostgreSQL
- 功能完善，SQL 过滤能力最强
- 前提是已有或愿意部署 PostgreSQL 实例
- 本项目设计为轻量级 Agent，不希望引入完整关系型数据库

## 四、为什么选择 Qdrant

### 1. 部署极简 —— 一条命令

```bash
docker run -d --name qdrant -p 6333:6333 -p 6334:6334 qdrant/qdrant
```

单容器、无依赖、镜像仅 180MB。

### 2. Go 客户端成熟可靠

`github.com/qdrant/go-client/qdrant`（135+ importers）：
- 基于 Protobuf 自动生成，与 Qdrant 服务端版本严格匹配
- 完整的 gRPC 接口，支持 upsert/search/scroll/filter 所有操作
- 活跃维护，发布节奏与 Qdrant 服务端一致
- 类型安全，编译期即可发现接口不匹配

### 3. 原生 HNSW 索引

Qdrant 的 HNSW 实现是用 **Rust** 编写的，性能优于 Weaviate 的 Go 实现。HNSW 是当前 ANN 搜索的最优算法之一，适合高维向量（1536 维）的近似最近邻搜索。

### 4. 内存效率

对于本项目当前规模（~10-12 个 chunk），Qdrant 的 HNSW 索引开销约 150MB，远低于 Milvus 的多层架构。即使扩展到 10 万级向量，Qdrant 的内存占用也在可控范围内。

### 5. 丰富的过滤能力

```go
// Qdrant 的 filter 支持：
filter := &qdrant.Filter{
    Should: []*qdrant.Condition{
        qdrant.NewMatch("doc_title", "RAG 技术介绍"),
        qdrant.NewMatch("doc_title", "Go 语言入门"),
    },
    Must: []*qdrant.Condition{
        qdrant.NewRange("score", &qdrant.Range{Gte: proto.Float64(0.5)}),
    },
}
```

支持 must/should/must_not 组合、范围查询、Geo 查询、文本匹配。

### 6. 持久化

Qdrant 将数据写入磁盘（默认 `/qdrant/storage`），支持 WAL（Write-Ahead Log）和快照备份。容器重启后数据自动恢复。

## 五、接入方案

### 架构变化

```
┌─────────────────┐        ┌─────────────────┐
│  内存 VectorStore │  ──→  │  Qdrant 客户端    │
│ (internal/rag)   │        │ (internal/rag)   │
│                  │        │         │        │
│ chunks []*Chunk  │        │   gRPC  │        │
│ 暴力 O(n) 搜索   │        │         ▼        │
└─────────────────┘        │  ┌─────────────┐  │
                            │  │  Qdrant 服务  │  │
                            │  │ (Docker)     │  │
                            │  └─────────────┘  │
                            └─────────────────┘
```

### 需要修改的文件

| 文件 | 改动 |
|------|------|
| `go.mod` | 新增 `github.com/qdrant/go-client/qdrant` |
| `internal/rag/rag.go` | 新增 `QdrantStore` 实现 `VectorStore` 接口 |
| `internal/config/config.go` | 可选：Qdrant 连接地址从环境变量读取 |
| `cmd/main.go` | 用 `QdrantStore` 替代 `NewVectorStore()` |
