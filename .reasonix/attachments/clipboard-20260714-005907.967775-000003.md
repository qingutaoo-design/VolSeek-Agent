## 1. 基础篇：为什么切片策略至关重要

### 面试题 1.1：什么是 RAG 中的文档切片（Chunking）？为什么它对 RAG 系统如此关键？

**参考答案：**

文档切片是将长文档分割成更小、更易于处理的文本片段（Chunks）的过程。在 RAG（检索增强生成）系统中，切片处于最底层的基础设施位置，直接影响上游所有环节：

```
原始文档 → 切片(Chunking) → Embedding → 向量存储 → 检索(Retrieval) → 生成(Generation)
                         ↑
                    这里是起点
```

**关键影响链路：**

```
切片质量 ↓ → Embedding 质量 ↓ → 检索精度 ↓ → 生成质量 ↓ → 用户感知 ↓
```

具体来说：

1. **切片过小（< 20 tokens）**：语义不完整，Embedding 向量无法捕捉有效语义，检索时产生大量"语义碎片噪音"
2. **切片过大（> 模型窗口）**：包含多主题信息，检索精度下降，且挤占 LLM 上下文窗口
3. **切片边界不合理**：关键信息被从中间切断，导致检索永远无法命中完整答案

> **面试追问：** 如果切片切得不好，会先影响哪个指标？
>
> 答：会先影响 **Recall（召回率）**。答案片段被切碎或切散后，向量检索无法匹配到完整语义单元。其次是 **Precision（精确率）**，切片过大引入无关噪音。最终导致 LLM **Faithfulness（忠实度）** 下降，因为没有足够完整的上下文支撑回答。

## 2. 策略篇：四种切片策略详解

### 面试题 2.1：请解释固定大小切片（Fixed-Size Chunking）的原理和优缺点

**参考答案：**

固定大小切片是最基础的策略，按固定字符/rune 数硬切，到尺寸就切一刀。

```go
// 核心逻辑
for start < totalLen {
    end := start + size
    chunk := text[start:end]
    start = end - overlap  // 带重叠滑动窗口
}
```

**优点：**
- 实现极简单，计算成本接近零
- 块大小高度可控，无超长块
- 总块数最少，存储和检索成本最低

**缺点：**
- **74% 的块在句子中间被截断**（实测数据），语义完整性最差
- 代码块、表格、列表等结构化内容被拦腰切断
- 同一主题内容可能分散在相邻块中

**适用场景：** 成本极度敏感的海量文档场景，或作为其他策略的兜底 fallback。

---

### 面试题 2.2：请解释句子边界切片（Sentence-Boundary Chunking）的原理

**参考答案：**

句子边界切片在固定窗口的基础上增加了"句子边界回溯"机制：在截断位置从后往前寻找句号、感叹号、问号或换行符等自然边界。

```go
// 核心逻辑：在截断附近找句子边界
if end < totalLen {
    for j := len(chunkRunes) - 1; j > size/2; j-- {
        if chunkRunes[j] == '。' || chunkRunes[j] == '.' || 
           chunkRunes[j] == '\n' || chunkRunes[j] == '！' || chunkRunes[j] == '？' {
            end = start + j + 1
            break
        }
    }
}
```

**关键设计要点：**
- 仅在非末尾块执行边界回溯（末尾块本就完整）
- 回溯范围限定在 `Size/2` 之后，防止块过短
- 使用 `[]rune` 而非 `[]byte`，正确处理中文标点

**实测效果：** 句子完整率从 26% 提升到 **65.7%**，提升约 40 个百分点。

---

### 面试题 2.3：请解释段落切片（Paragraph Chunking）和递归切片（Recursive Chunking）的原理和区别

**参考答案：**

| 维度 | 段落切片 | 递归切片 |
|------|---------|---------|
| **核心思想** | 以空行分隔的段落为基本单位，优先保持段落完整 | 按分隔符优先级逐级尝试：段落→行→句子→字符 |
| **实现方式** | 贪心合并段落，单段落超长时降级到句子边界切分 | 递归拆分，每级用不同粒度分隔符 |
| **分隔符优先级** | 空行段落 > 句子 | `\n\n` > `\n` > `。` > `.` > `！` > `？` > `!` > 字符 |
| **块大小控制** | 中等，段落大小不均时波动较大 | 较弱，容易产生大量碎片 |

**段落切片的核心逻辑：**

```
按 \n\n 拆分段落实体 → 尝试合并段落至 Size 内 → 
    ├─ 单个段落 > Size → 降级到句子边界切分
    └─ 合并后 > Size → flush 当前块，开始新块
```

**递归切片的核心逻辑（类似 LangChain RecursiveCharacterTextSplitter）：**

```
function split(text, separators=[\n\n, \n, 。, ., ！, ？, !, ""], depth):
    if len(text) <= Size: return [text]
    parts = split_by(separators[depth])
    if len(parts) == 1: return split(text, depth+1)  // 换更细的分隔符
    贪心合并 parts 至 Size 内
    对超长 part 递归调用 split(part, depth+1)
```

> **面试追问：** 递归切片的碎片率为什么高？
>
> 答：递归切片使用单字符（`。` `！` `？`）作为分隔符时，会把每个句子甚至标点两侧都拆成独立片段。如果合并逻辑不够智能（例如句子太短不满足合并条件），就会产生大量 < 20 runes 的碎片。实测递归切片在长文档中碎片率达 **26.7%**。

## 3. 评测篇：四层递进检测体系

### 面试题 3.1：请系统性地介绍 RAG 切片策略的四层评测体系

**参考答案：**

四层评测体系从底层到上层逐层递进，覆盖切片质量的全链路影响：

```
┌─────────────────────────────────────────────────────────┐
│  Layer 4: 工程成本                                       │
│  存储大小 / Token消耗 / 检索时延 / 预处理耗时              │
├─────────────────────────────────────────────────────────┤
│  Layer 3: 端到端生成                                      │
│  忠实度(Faithfulness) / 相关性(Relevance) / 准确率         │
│  (需要 LLM-as-Judge 评测)                                 │
├─────────────────────────────────────────────────────────┤
│  Layer 2: 检索量化                                        │
│  Recall@K / Precision@K / MRR / nDCG / 噪声率            │
│  (需要标注 QA 测试集 + Embedding + 向量检索)               │
├─────────────────────────────────────────────────────────┤
│  Layer 1: 静态块质量 (最基础, 切完即可自检)                 │
│  块大小 / 标准差 / 碎片率 / 句子完整率 / 代码块完整性        │
└─────────────────────────────────────────────────────────┘
```

**设计原则：**
1. **下层是上层的必要但不充分条件**：Layer 1 好不一定 Layer 2 好，但 Layer 1 差 Layer 2 一定差
2. **逐层递进，成本递增**：Layer 1 无外部依赖，几毫秒出结果；Layer 2/3 需要 Embedding API 和 LLM 调用
3. **分层解耦**：每层可独立运行，POC 阶段可只跑 Layer 1

---

### 面试题 3.2：Layer 1 静态块质量评测包含哪些具体指标？如何解读？

**参考答案：**

Layer 1 是所有评测的基础，包含**三类指标**：

#### 3.2.1 基础统计指标

| 指标 | 计算公式 | 解读 |
|------|---------|------|
| 总块数 | `len(chunks)` | 越少存储成本越低 |
| 平均块大小 | `sum(runes) / n` | 应接近目标 Size |
| **标准差** | `sqrt(Σ(x-μ)²/n)` | **越小越均匀**，太大说明块大小失控 |
| 碎片率 | `< 20 runes 的块数 / 总块数` | **应趋近于 0**，碎片丢失语义 |
| 重叠率 | `重叠字符 / 文档总长` | 10-20% 合理，过高浪费 token |

#### 3.2.2 语义完整性指标（自动计算，无需 LLM）

| 指标 | 检测方法 | 解读 |
|------|---------|------|
| **句子完整率** | 块末尾字符是否是 `。.!！？\n` | **核心指标**，低于 50% 说明大量截断 |
| 代码块切断数 | 检测 ` ``` ` 是否成对 | 切断代码块 = 不可编译的代码 |
| 段落起始保留 | 块首是否是标题/列表标记 | 影响上下文可读性 |

#### 3.2.3 LLM 内聚度评分（可选，需 LLM 调用）

对每个 chunk 用 LLM 打分（1-5）：
- **Intra-chunk Cohesion**：同一块内话题统一度
- **Block Integrity**：完整语义单元未被拆分
- **Size Compliance**：是否在窗口限制内

---

### 面试题 3.3：Layer 2 检索量化评测如何实现？核心指标有哪些？

**参考答案：**

Layer 2 需要三个前置条件：
1. **标注 QA 测试集**：`{question, ground_truth, source_doc}`
2. **Embedding 模型**：对 query 和 chunks 做向量化
3. **向量检索逻辑**：余弦相似度 Top-K

**实现流程：**

```
┌──────────┐    ┌──────────┐    ┌────────────┐
│  QA 集   │    │ 向量库   │    │ 检索评测    │
│ 16道问题  │    │ Top-K    │    │ Recall/MRR │
└────┬─────┘    └────┬─────┘    └──────┬──────┘
     │               │                 │
     ▼               ▼                 ▼
  query= "goroutine是什么?"            
     │               ▲                 
     └───embedding───┘  ┌─────────────────┐
                        │ cosine(query,    │
                        │ chunk_i) 排序    │
                        └─────────────────┘
```

**核心指标：**

| 指标 | 公式 | 含义 | 生产标准 |
|------|------|------|---------|
| **Recall@K** | `命中答案的问题数 / 总问题数` | 答案是否被召回 | @5 ≥ 90% |
| **Precision@K** | `相关块数 / 返回块数` | 返回结果中噪音比例 | ≥ 70% |
| **MRR** | `mean(1/rank)` | 答案排得有多靠前 | ≥ 0.8 |
| **噪声率** | `无关块数 / Top-K` | 检索结果纯度 | ≤ 30% |

---

### 面试题 3.4：Layer 3 端到端生成评测和 Layer 4 工程成本评测分别关注什么？

**参考答案：**

#### Layer 3：端到端生成评测

模拟真实 RAG 流程：检索上下文 → 注入 LLM → 生成回答 → 评估质量。

```
对每个 QA 问题:
  1. 从切片结果中定位答案所在 chunk
  2. 取答案块 + 前后邻接块作为"检索上下文"(模拟 Top-3)
  3. 将上下文 + 问题喂给 LLM 生成回答
  4. 用 LLM-as-Judge 从 3 个维度打分(1-5):
     - Faithfulness（忠实度）：回答是否基于上下文，不编造
     - Relevance（相关性）：回答是否切题
     - Correctness（准确率）：与标准答案的一致性
```

**典型切片问题的评测表现：**

| 切片问题 | Layer 2 表现 | Layer 3 表现 |
|---------|-------------|-------------|
| 切片过小 | Recall ↓ | 忠实度 ↓，大量幻觉 |
| 切片过大 | Precision ↓ | 上下文噪声高，混淆 LLM |
| 无分层 | 跨章节问题 Recall 暴跌 | 缺少关键上下文 |

#### Layer 4：工程成本评测

| 成本项 | 计算方式 | 策略间差异 |
|-------|---------|-----------|
| 向量存储 | `块数 × 维度 × 4 bytes` | fixed-size 最低(464KB)，recursive 最高(848KB) |
| Embedding Token | `总 runes / 1.5` | 差异约 10%，受碎片影响 |
| 单次查询 Token | `Top-K 块的 runes / 1.5` | 越大成本越高，paragraph 约多 30% |
| 超限块数 | `> 2500 runes 的块数` | 所有策略均为 0（测试范围内） |

## 4. 指标篇：核心指标解读

### 面试题 4.1：Recall@K 和 Precision@K 为什么对切片策略敏感？用"鱼网捕鱼"的类比解释。

**参考答案：**

把切片策略想象成**制作渔网的过程**：

```
固定大小切片 = 统一尺寸的渔网
句子边界切片 = 顺着鱼群轮廓剪裁的网
段落切片     = 按大鱼群区域分格的网
递归切片     = 多级网眼的网
```

- **网眼太小（切片过碎）**：`[Go语言][是由][Google][开发的]` → 每条鱼（完整语义）都漏掉了 → **Recall 低**
- **网眼太大（切片过大）**：`[Go语言是由Google开发的一种静态强类型语言，同时Docker也是用Go写的...]` → 一网捞起很多无关内容 → **Precision 低**
- **网孔刚好对准鱼群**：`[Go语言是由Google开发的一种静态强类型语言]` → Recall 和 Precision 同时高

**数学本质：** 切片策略决定了 Embedding 向量的"信息密度分布"。好的切片让每个向量捕捉一个主题，差的切片让向量要么信息不足（碎片），要么信息混杂（过大）。

---

### 面试题 4.2：句子完整率为什么是 Layer 1 最重要的指标？它与 Recall 的关联是什么？

**参考答案：**

句子完整率衡量的是"块末尾字符是否是句子结束符"的比例。

**为什么重要？** 因为 Embedding 模型的工作方式是**对整个向量做语义压缩**。如果一个块在句子中间截断：

```
原文: "Goroutine是一种轻量级线程，由Go运行时管理，创建成本极低。"
截断: "Goroutine是一种轻量级线程，由Go运行时管理，"
```

这个截断块的向量语义可能是"Go运行时管理了某种东西"，而不是"Goroutine是轻量级线程"这个完整语义。

**实地数据：**

| 策略 | 句子完整率 | 对应的召回表现（预期） |
|------|:---------:|:-------------------:|
| fixed-size | 26.0% | 基准 |
| sentence-boundary | 65.7% | ↑ 预期提升 2-3x |
| paragraph | 65.8% | ↑ 预期接近 |
| recursive | 60.3% | ↑ 但碎片拖累 |

> 注意：句子完整率从 26% 到 66% 是一个 **2.5 倍提升**，这在检索场景下通常能转化为显著的 Recall 提升。

### 面试题 5.2：什么情况下 fixed-size 策略反而比高级策略好？

**参考答案：**

在以下三种情况下，fixed-size 可能是最优选择：

**1. 无标点的连续文本**
```
原文："根据最新数据显示全球经济增速放缓通货膨胀压力持续..."
```
当文本完全没有句号、换行等自然边界时，`sentence-boundary` 的回溯机制永远不会触发，结果与 fixed-size 完全相同。测试中的 `nopunct.txt` 即为这种场景，4 种策略结果无差异。

**2. 成本极度敏感的海量场景**
- 假设有 1000 万文档
- fixed-size: 1.16 亿向量 → 约 4.6 GB 存储
- sentence-boundary: 1.41 亿向量 → 约 5.6 GB 存储（+22%）
- paragraph: 2.08 亿向量 → 约 8.3 GB 存储（+79%）
- 成本差异在百万级文档场景下非常显著

**3. 短文档为主（文档长度 < Size）**
对于短文本，所有策略等价（都作为整块返回）。只有当文档长度超过 Size 时，策略差异才开始体现。

## 6. 决策篇：生产环境策略选型指南

### 面试题 6.1：面对不同文档类型，如何为生产环境选择切片策略？给出决策树。

**参考答案：**

```
文档特征分析
│
├─ ❶ 文档长度: 是否绝大多数 < chunk_size?
│   └─ 是 → 策略差异可忽略，选最简单的 fixed-size
│
├─ ❷ 文档结构: 是否包含 Markdown / 代码 / 表格?
│   ├─ 纯文本有声调 → sentence-boundary
│   ├─ Markdown 层次丰富 → paragraph（保留标题结构更好） 
│   └─ 代码密集 → sentence-boundary（代码块切断最少）
│
├─ ❸ 检索精度要求: Recall@5 是否需要 ≥ 90%?
│   ├─ 是 → 优先 sentence-boundary 或 paragraph
│   └─ 否 → 可考虑成本更低的 fixed-size
│
├─ ❹ 成本约束: 文档量级?
│   ├─ < 10万 → 用最好的策略，不必纠结成本
│   ├─ 10万~100万 → sentence-boundary（最佳性价比）
│   └─ > 1000万 → 考虑两层策略：fixed-size + 检索后拼接
│
└─ ❺ 最终推荐:
    ├─ 🥇 sentence-boundary → 通用首选
    ├─ 🥈 paragraph → 结构化文档
    ├─ 🥉 fixed-size → 极限成本敏感
    └─ ⚠ recursive → 需配合碎片后处理
```

### 面试题 6.2：什么是"切片策略的边际收益递减效应"？请解释。

**参考答案：**

切片策略对 RAG 系统的贡献存在明显的边际收益递减曲线：

```
RAG 系统总质量
    ↑
100%│                                                ┌── recursive(+83%成本)
    │                                          ┌─────┤
 80%│                                   ┌──────┤     └── paragraph(+79%成本)
    │                             ┌─────┤      └── sentence-boundary(+22%成本)
 60%│                       ┌─────┤    
    │                  ┌────┘     
 40%│            ┌─────┘          
    │       ┌────┘
 20%│  ┌────┘
    │  │ fixed-size(基准)
  0% └──┴──┴──┴──┴──┴──┴──┴──→ 切片策略复杂度/成本
```

**规律发现：**

1. **从 fixed-size 到 sentence-boundary**：+22% 的成本换来 **约 40 个百分点的句子完整率提升**——性价比最高
2. **从 sentence-boundary 到 paragraph**：+57% 的额外成本（79%-22%）但句子完整率几乎不变（65.7%→65.8%）——边际收益接近于零
3. **从 paragraph 到 recursive**：成本继续攀升，但引入了碎片问题——甚至可能产生负收益

**结论：** `sentence-boundary` 位于边际收益曲线的"拐点"附近，是投入产出比最优的选择。

### 面试题 6.3：你们项目当前的切片实现有什么可以改进的地方？请从工程角度提出建议。

**参考答案：**

基于对 VolSeek RAG Agent 的 `rag.go` 源码分析，提出以下改进建议：

#### 改进 1：Markdown 检测逻辑优化

**当前问题：**
```go
if strings.Contains(text, "# ") || strings.Contains(text, "## ") {
    return c.chunkByMarkdown(text, title)
}
```
简单的字符串包含检测，`"C#是一种语言"` 会被误判为 Markdown。

**建议：**
```go
// 只检测行首的 # 标记
lines := strings.Split(text, "\n")
hasMarkdown := false
for _, line := range lines {
    if strings.HasPrefix(strings.TrimSpace(line), "# ") {
        hasMarkdown = true
        break
    }
}
```

#### 改进 2：增加分块后处理——碎片合并

**当前问题：** recursive 策略产生大量 < 20 runes 的碎片块。

**建议：** 在 `Chunk()` 输出后增加一个合并 pass：
```go
func mergeFragments(chunks []*Chunk, minSize int) []*Chunk {
    var merged []*Chunk
    for i, ch := range chunks {
        if len([]rune(ch.Content)) < minSize && len(merged) > 0 {
            // 碎片合并到前一个块
            merged[len(merged)-1].Content += "\n" + ch.Content
        } else {
            merged = append(merged, ch)
        }
    }
    return merged
}
```

#### 改进 3：重叠参数的动态调整

**当前问题：** Overlap 是固定值，对不同长度文档行为不一致。

**建议：** 动态重叠：
```go
// 短文档：重叠比例加大
if docRuneLen < size*3 {
    overlap = size / 3  // 33%
} else {
    overlap = size / 10  // 10%
}
```
