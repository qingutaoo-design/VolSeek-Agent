# 面试问答：RAG Agent 架构设计

> 基于 VolSeek-Agent 项目的 Plan-Execute-Reflect 架构与普通 ReAct Agent 的对比分析

---

## Q1：这个 Agent 和你见过的普通 ReAct Agent 有什么区别？你为什么要做这种设计？

**🎯 回答方向**：先点出最直观的不同，再展开到架构层面的差异。

最直观的区别是执行模式。普通 ReAct Agent 是**边想边做**——LLM 生成一句思考、调用一个工具、观察结果、再思考下一步，循环直到结束。VolSeek-Agent 是**先规划后执行**——接到问题后第一步不是检索，而是先生成一个结构化计划，再按计划执行。

从回答质量上看，区别体现在四个层面：

| 维度 | 普通 ReAct Agent | VolSeek-Agent |
|------|-----------------|---------------|
| **执行模式** | 边想边做（ReAct 循环） | 先规划后执行（Plan → Execute 分离） |
| **检索策略** | 简单向量搜索 | 7 种意图路由 + 知识图谱 |
| **工具调度** | 串行执行 | 并行工具调用 |
| **质量控制** | 一次生成直接返回 | 自我反思评审后修正 |

## Q2：你做的 Agent 如果知识库里没有相关内容，为什么它还是能回答准确？

**🎯 回答方向**：澄清一个常见误解——RAG 是"增强"不是"替代"。

这是一个很有价值的观察。如果知识库里没有"法国的首都"相关资料，但 Agent 仍然回答"巴黎"，这不是 bug，而是 RAG 架构的**设计特性**。

核心原因分三层：

**第一层：检索返回空，但 LLM 自己有知识**

系统在 `rag.go:502` 设置了检索门槛 `score >= threshold（0.5）`。如果知识库里所有 chunk 与问题的 cosine similarity 都低于 0.5，`knowledge_search` 工具就返回空数组。

这个空结果作为 tool response 回给 LLM 后，LLM 不会傻掉——它自己在训练时已经学过「法国首都=巴黎」。它直接调用 `final_answer` 或者自然 stop 给出答案。

RAG 的全称是 Retrieval-**Augmented** Generation——检索是"增强"不是"替代"。LLM 本身就有知识，RAG 只是给它补充上下文。

**第二层：LLM 自然停止也能给出答案（engine.go:209）**

```go
} else if resp.FinishReason == "stop" {
    if resp.Content != "" {
        state.FinalAnswer = resp.Content
        state.Confidence = 0.7
    }
}
```

即使没调用 `final_answer` 工具，LLM 自然停止时也会直接输出内容。

**第三层：反思阶段还能再修（engine.go:231）**

```go
if e.config.EnableReflection && state.FinalAnswer != "" {
    refinedAnswer, confidence, _ := e.reflectOnAnswer(...)
    state.FinalAnswer = refinedAnswer
}
```

生成答案后再问一次 LLM "这个答案对不对？"，给了第二次纠正机会。

## Q3：那这和普通 Agent 有什么区别？普通 Agent 不也能用 LLM 自己的知识回答吗？

**🎯 回答方向**：诚实承认在"知识库外回答"这件事上没有区别，然后转折到真正有区别的地方。

你说得对，**在"回答知识库外的问题"这件事上，VolSeek-Agent 和普通 ReAct Agent 确实没有本质区别**。都是检索不到 → LLM 用自己的训练知识兜底。

真正的区别不在这里，而在**当知识库里确实有相关内容时，系统怎么做**。

### 区别 1：执行模式 —— 先规划再执行

普通 ReAct Agent 是：

```
问题 → 检索 → LLM 生成 → 返回
```

VolSeek-Agent 是：

```
问题 → 意图分析 → 制定计划 → 按计划逐步执行 → 反思 → 返回
```

先规划的优势在于：Agent 在动手之前就知道自己要做什么，不会在检索过程中跑偏。比如用户问"比较 RAG 和 GraphRAG"，普通 Agent 检索一次发现不够就再检一次，可能连续检四五次；VolSeek-Agent 的 Plan 阶段就确定了"这是一个比较性查询，需要混合检索 + RRF 融合，检两次就够了"。

### 区别 2：检索策略 —— 7 种意图路由（rag.go:563-611）

```go
switch intent.Type {
case QueryFactual:      → 关键词搜索（精确匹配）
case QueryConceptual:   → 纯向量搜索（语义匹配）
case QueryComparative:  → 混合搜索 + RRF 融合排序
case QueryAnalytical:   → 向量搜索 + 知识图谱拓展
case QueryRecent:       → 不检索（需要 web_search）
}
```

普通 Agent 不管什么问题都走同一个检索路径。VolSeek-Agent 先用 LLM 分析查询意图，7 种类型各走不同的检索策略——事实性问题用关键词精确匹配，概念性问题用语义向量，比较性问题做混合搜索再加权融合。

### 区别 3：工具调度 —— 并行执行

普通 Agent 串行调用工具，一个超时可能卡住整个流程。VolSeek-Agent 用 `sync.WaitGroup` 做并行工具调用，互不依赖的工具同时执行。

### 区别 4：质量控制 —— 自我反思

普通 Agent 一次生成就直接返回，答案质量全靠 LLM 的一次发挥。VolSeek-Agent 生成答案后还有一个反思阶段，从**准确性、完整性、清晰度**三个维度评审，并自动修正。

## Q4：如果反思阶段发现答案有问题，它能怎么修？有没有什么它修不了的情况？

**🎯 回答方向**：坦诚回答局限性，展示思考深度。

它能修的是**推理逻辑问题**——比如 LLM 从上下文得出了错误的结论，反思阶段让它重新审视一遍，可能纠正。它修不了的是**切片导致的知识丢失**。

这一点很关键：`reflectOnAnswer(ctx, query, answer)` 的输入只有问题和答案，没有原始检索到的 chunk 内容（agent/engine.go:234）：

```go
refinedAnswer, confidence, _ := e.reflectOnAnswer(ctx, query, state.FinalAnswer)
```

如果切片策略把"规划阶段的意图分析"和"执行阶段的工具列表"切到两个 chunk 里，Agent 回答问题的时候只看到了半个上下文，导致回答不完整——反思阶段**不知道 chunk 里丢了什么**，它只能凭 LLM 自身对相关知识的了解来判断"这个答案对不对"。切片造成的知识缺失，反思阶段兜不住。

---

## Q5：那你觉得这个架构最大的设计败笔是什么？

**🎯 回答方向**：有自信指出不足

最大的问题是**检索失败的反馈不够明显**。如果 `knowledge_search` 返回空，Agent 得到的结果和知识库里恰好没相关内容的是一模一样的空数组。Agent 无法区分"搜索执行成功了但确实没有"和"搜索执行出错了"。

另一个问题是**反思阶段不感知检索结果**。反思只看最终答案，不看原始 chunk——这意味着系统无法判断"这个答案是基于知识库事实还是基于 LLM 幻觉"。如果能把检索到的 chunk 摘要也传入反思阶段，反思质量会更高。

## Q6：如果让你继续优化这个架构，你会从哪里入手？

三个方向：

1. **检索置信度透传到反思**：让反思阶段知道当前答案有多少来自知识库的事实、多少来自 LLM 自身知识。如果来自 LLM 自身知识的比例过高，降低置信度。

2. **动态规划重试**：目前 Plan 阶段做一次规划就不再改了。如果 Execute 阶段发现检索结果和预期不符，应该能回到 Plan 阶段重新规划。目前的设计是 Plan → Execute → Reflect 单向流水线，没有回溯机制。

3. **检索失败时的显式提示**：当 `knowledge_search` 返回空或 low score 时，在 tool result 里显式提示"知识库中没有相关内容"，让 LLM 至少知道自己是在凭训练知识回答，而不是从上下文里找答案。
