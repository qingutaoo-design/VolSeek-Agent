# 面试问答：Agent 执行引擎深度解析

> 基于 VolSeek-Agent 项目的 Plan-Execute-Reflect 架构源码分析

---

## Q1：这个 Agent 的"子任务逐个执行"是怎么实现的？我没看懂你的源码。

**🎯 回答方向**：先破除一个常见的理解误区——子任务不是"按列表逐一执行"的。

很多人看到 `plan.SubGoals` 列表，以为代码会像这样运行：

```go
for _, subGoal := range plan.SubGoals {
    executeSubGoal(subGoal)  // ❌ 不是这样
}
```

实际上，子任务列表只是一个**纯文本提示**，真正驱动执行的是一个 **Think → Act → Observe 循环**，和 ReAct 一样。看核心循环（`engine.go:129-226`）：

```go
for stepIndex < maxRounds {    // 最多 10 轮
    // 每轮：LLM 看对话历史，决定下一步
    resp, _ := e.llm.ChatWithRetry(ctx, messages, openAITools, 0.7)

    if len(resp.ToolCalls) > 0 {
        for _, tc := range resp.ToolCalls {
            if tc.Name == "final_answer" {
                needFinalAnswer = true  // 提交最终答案 → 结束
                break
            }
        }
        if !needFinalAnswer {
            // 执行工具，结果追加到 messages
            results := e.executeToolCalls(ctx, resp.ToolCalls, ...)
            messages = append(messages, toolResultMessages...)
            continue  // 下一轮循环
        }
    } else if resp.FinishReason == "stop" {
        state.FinalAnswer = resp.Content  // LLM 自然停止
        needFinalAnswer = true
    }
    if needFinalAnswer { break }
}
```

**每一轮循环**：
1. LLM 看当前对话历史（包括之前所有工具结果）
2. 自己决定下一步做什么——调工具还是给答案
3. 执行并记录，回到步骤 1

这就是"子任务逐个进行"的实现。LLM 自己决定顺序，代码不给它下指令。

## Q2：那 `plan.SubGoals` 到底有什么用？如果执行时不按它来，不是白做了吗？

**🎯 回答方向**：承认它确实不是强约束，但解释它的实际用途——作为 system prompt 中的参考信息。

看 `createPlan`（`engine.go:285-295`）：

```go
planPrompt := `Analyze the question and create a structured execution plan.
Available tools: knowledge_search, graph_search, calculator, final_answer

Respond with JSON:
{
  "goal": "...",
  "sub_goals": ["Search for RAG", "Search for GraphRAG", "Compare"],
  "requires_tool": true/false,
  "reasoning": "..."
}`
resp, _ := e.llm.Chat(ctx, messages, nil, 0.3)  // 不带工具，低温度
```

这个 JSON 计划被嵌入到 system prompt 里（`engine.go:113`）：

```go
systemPrompt := e.buildSystemPrompt(ctx, query, plan)
```

所以子目标列表只是以**纯文本形式**放在 system prompt 中，给 LLM 做参考。LLM 可以选择遵从也可以不遵从。

**那它有什么用？**

核心作用是**让 LLM 在动手之前先想一遍**。传统 ReAct 第一次调用 LLM 时直接带工具列表，LLM 可能还没想清楚就乱调工具。Plan 阶段的调用是 `Chat(ctx, messages, nil, 0.3)`——不带工具、低温度，LLM 只能输出文本，强制它先理清思路。

这份思路接着出现在 system prompt 里，相当于给了 LLM 一张"地图"。虽然它执行时可以不走地图，但看过地图再走，比直接盲走更容易走对方向。

## Q3：你说的"不带工具"具体指什么？怎么做到的？

**🎯 回答方向**：指向具体的 API 调用差异。

看第 302 行：

```go
// Plan 阶段：不传工具
resp, err := e.llm.Chat(ctx, messages, nil, 0.3)
//                                     ↑   ↑
//                              openAITools=nil  temperature=0.3
```

再看第 140 行执行循环：

```go
// Execute 阶段：传工具
resp, err := e.llm.ChatWithRetry(ctx, messages, openAITools, 0.7)
//                                          ↑            ↑
//                                   openAITools≠nil   temperature=0.7
```

差异在两点：

| 参数 | Plan 阶段 | Execute 阶段 |
|------|---------|-------------|
| `tools` | `nil`（LLM 无法调工具） | `openAITools`（LLM 可以调） |
| `temperature` | 0.3（计划要稳定） | 0.7（执行要有创造性） |

当 `tools=nil` 时，OpenAI/DeepSeek 的 API 不会返回 `tool_calls`，LLM 只能输出文本——强制它"纯思考"。这就是 Plan 阶段和 ReAct 最大区别的技术实现。

## Q4：那 Execute 阶段和传统 ReAct 有什么区别？是不是完全一样？

**🎯 回答方向**：承认核心循环一样，但指出两个差异——Plan 的上下文影响 + 并行工具调用。

核心循环结构确实一样（Think → Act → Observe），区别在于：

### 区别 1：系统提示词含计划

传统 ReAct 的 system prompt 只有角色定义：

```
You are a helpful assistant with access to the following tools...
```

VolSeek-Agent 的 system prompt 额外包含了计划：

```
You are a helpful assistant...
Your plan:
  Goal: Compare RAG and GraphRAG
  Sub-goals: 1. Search RAG info 2. Search GraphRAG info 3. Compare
```

这会影响 LLM 的行为——它更容易做出符合计划的决策，而不是随机调工具。

### 区别 2：并行工具调用（`engine.go:365-371`）

```go
if e.config.ParallelToolCalls && len(calls) > 1 {
    e.executeParallel(ctx, calls, results, ch, stepIndex)
} else {
    for i, tc := range calls {
        results[i] = e.executeSingle(ctx, tc, ch, stepIndex)
    }
}
```

传统 ReAct 串行执行工具。如果 LLM 一次调了两个工具，先执行第一个、看结果、再执行第二个。VolSeek-Agent 可以用 goroutine 同时执行多个独立工具：

```go
func (e *AgentEngine) executeParallel(...) {
    var wg sync.WaitGroup
    for i, tc := range calls {
        wg.Add(1)
        go func() {
            defer wg.Done()
            results[i] = e.executeSingle(ctx, tc, ...)  // 并发
        }()
    }
    wg.Wait()  // 所有完成后再继续
}
```

这意味着 `knowledge_search` 和 `graph_search` 可以同时调，节约一半时间。

## Q5：如果 LLM 在 Execute 循环里跑偏了——不按计划来、反复调同一个工具怎么办？

**🎯 回答方向**：三个保护机制——最大轮次、final_answer 工具、反思阶段。

### 保护 1：`MaxPlanningRounds`（`engine.go:124-127`）

```go
maxRounds := e.config.MaxPlanningRounds
if maxRounds <= 0 {
    maxRounds = 10  // 默认最多 10 轮
}
for stepIndex < maxRounds { ... }
```

不管 LLM 怎么跑偏，最多转 10 圈就必须结束。

### 保护 2：`final_answer` 工具强制结束

```go
if tc.Name == "final_answer" {
    needFinalAnswer = true
    break  // 一旦 LLM 调了这个工具，不管还有没有其他工具调用，直接结束
}
```

`final_answer` 是**唯一一个 LLM 必须显式调用的工具**来提交答案（不像其他 Agent 那样 LLM 自然 stop 就算结束）。这确保 LLM 不会"忘记"给答案。

### 保护 3：反思阶段修正（`engine.go:231-239`）

```go
if e.config.EnableReflection && state.FinalAnswer != "" {
    refinedAnswer, confidence, _ := e.reflectOnAnswer(ctx, query, state.FinalAnswer)
    state.FinalAnswer = refinedAnswer  // 用修正后的替换
}
```

即使执行阶段跑偏了导致答案质量差，反思阶段还能拉回来一次。

## Q6：你觉得这个架构最大的问题是什么？

**🎯 回答方向**：指出不足比只说好话更显深度。

最大的问题是**Plan 阶段没有回溯机制**。当前设计是 Plan → Execute → Reflect 单向流水线：

```
Plan → Execute → Reflect
```

Plan 阶段做一次计划就不再改了。如果 Execute 阶段发现检索结果和 Plan 的假设不符（比如计划说"搜索 A 然后搜索 B"，但实际上 A 的结果直接给出了完整答案），系统不会回到 Plan 阶段重新规划。

更理想的方案是 Plan-Execute 之间可以双向循环：

```
Plan ←→ Execute → Reflect
```

当 Execute 发现需要调整计划时，能回到 Plan 阶段重新规划。这个能力叫 **RePlanning**，当前版本没有实现。
