# 面试问答：Agent 记忆系统设计

> 基于 VolSeek-Agent 项目的三级记忆架构设计

---

## Q1：你给 Agent 设计的记忆系统是什么样的？说说架构。

参照人类记忆模型，我设计了三层架构：

```
┌──────────────────────────────────────────┐
│  3. 语义记忆 (RedisMemory)                │
│     跨会话持久化，存储关键事实、用户偏好     │
│     Redis List + TTL 自动过期              │
├──────────────────────────────────────────┤
│  2. 情节记忆 (BufferMemory)               │
│     单会话内最近 N 轮对话的环形缓冲区        │
│     内存存储，无持久化                      │
├──────────────────────────────────────────┤
│  1. 工作记忆 (messages []LLMMessage)       │
│     当前轮次的 LLM context window           │
│     agent 原有的 messages 切片              │
└──────────────────────────────────────────┘
```

---

## Q2：为什么选择了这种三层设计？为什么不用一层解决所有问题？

用性能/成本/容量的三角平衡来解释：

| 记忆层 | 容量 | 速度 | 持久化 | 成本 |
|--------|------|------|--------|------|
| 工作记忆 | KB 级 | 纳秒 | 无（进程内） | 0 |
| 情节记忆 | MB 级 | 纳秒 | 可选（内存文件） | 接近 0 |
| 语义记忆 | GB 级 | 毫秒 | 持久化 | Redis 开销 |

单层方案的问题：
- 全部扔给 LLM context window：超出 Token 限制，成本高
- 全部用 Redis：每次都要网络 IO，简单场景杀鸡用牛刀
- 全部用内存文件：没有过期机制，重启全丢

---

## Q3：为什么引入 Redis？用内存文件不好吗？

优缺点都要摆出来：

### 为什么用 Redis

| 需求 | 内存文件 | Redis |
|------|---------|-------|
| 读速度 | 纳秒（已加载） | 毫秒 |
| 跨进程共享 | 不支持 | 支持 |
| TTL 自动过期 | 自己写 cron | 原生支持 |
| 数据结构 | 自己管理 | List/Hash/SortedSet |
| 持久化 | 自己写 JSON dump | AOF + RDB |

Redis 的核心优势是数据结构丰富 + TTL 内置。对话记忆天然适合用 List 存，加上 TTL 自动老化。

### 坏处与缓解

| 坏处 | 缓解方案 |
|------|---------|
| 多一个依赖 | `docker run` 一条命令，镜像 30MB |
| 网络延迟 | 单命令 0.5ms，一条对话共 1ms |
| 重启丢数据 | 配置 AOF + RDB |
| 内存成本 | TTL 控制，7 天过期 |

### 最终设计：BufferMemory 兜底 + Redis 可选

```go
mem := memory.NewBuffer(100)
redisAddr := config.GetEnv("REDIS_ADDR", "")
if redisAddr != "" {
    redisMem, _ := memory.NewRedis(redisAddr, ...)
    mem = redisMem
}
```

不配置 Redis -> 纯内存，零依赖。配置了 -> 自动升级为持久化。

## Q4：记忆怎么保存的？每次回答完都存吗？

在 executeWithEvents 末尾：

```go
e.memory.Save(ctx, "default", memory.Entry{Role: "user", Content: query})
e.memory.Save(ctx, "default", memory.Entry{Role: "assistant", Content: answer})
```

频率：每轮对话存 2 条。BufferMemory 是纯内存 append，Redis 是 LPUSH + EXPIRE，0.5ms 以内。
