# 面试问答：RAG 系统的文档切片策略优化

> 基于 VolSeek-Agent 项目的 Chunker 模块重构经验整理

---

## Q1：你在这个 RAG Agent 项目中负责了哪些模块？

**🎯 回答方向**：先定位自己负责的范围，再引出切片策略这个亮点。

我主要负责 RAG 引擎的**文档切片（Chunker）模块**的重构。原有系统只有两种切片策略——按 Markdown 标题分和固定窗口滑动分，存在几个明显的语义断层问题。我重新设计了一套**四级流水线架构**，把切片质量提升了一个档次。

---

## Q2：你为什么要重写切片策略？原来的方案有什么问题？

**🎯 回答方向**：先展示你对原方案的理解，再点出具体的痛点。要举具体的例子。

原有的设计思路是「按文本类型二选一」：

- **Markdown 文本** → 遍历每行，遇到标题就切一块，行数累计到 `Size（500字符）` 就硬切
- **纯文本** → 固定窗口滑动 + 句子边界截断

听起来合理，但实际跑起来有三个核心问题：

### 问题一：Markdown 大章节被拦腰截断

比如一个 `## 核心算法` 标题下有 2000 字，旧代码的逻辑是逐行累加，到 500 字符就 flush 一次。这个 500 字符的边界**不是语义边界**——可能在「注意力机制是…」的中间就切断了。下一块又接着写，但上一块的尾部内容和下一块的头部内容之间毫无关联。

### 问题二：纯文本不感知段落

纯文本模式下，代码只找 `。！？` 这些句子边界，但**完全不识别 `\n\n`（段落空行）**。比如一段技术文档：

```
RAG 的核心思想是检索+生成。

GraphRAG 在此基础上增加了知识图谱。
```

这两句是不同段落，语义相对独立。但旧方案会直接把它们混在一个 chunk 里，或者恰好从中间切断，导致检索时匹配到的内容语义混杂。

### 问题三：没有碎片防护

旧方案没有 `MinSize`，一个 30 字的短句可能自成一 chunk。向量化后，这个 chunk 的语义信息太少，检索时 cosine similarity 几乎被噪声淹没。

---

## Q3：那你具体是怎么优化的？讲讲你的设计方案。

**🎯 回答方向**：讲清楚架构设计，用「流水线」这个比喻帮助理解。

我的核心思路是：**不要一刀切，要分级降级处理**。我构建了一条**四级流水线**：

```
原始文本
    │
    ▼
第一级：段落感知分割（splitByParagraph）
    │  按 \n\n 空行拆成段落，保证每段语义完整
    │
    ▼
第二级：段落大小判断
    │  ├── ≤ Size → 直接成块（✅ 最优路径）
    │  └── > Size → 进入递归切割（splitOversized）
    │
    ▼
第三级：递归句子切割
    │  取 Size 范围，用 findBestBoundary 找最佳截断点
    │  优先级：段落空行 > 句尾 > 换行 > 标点
    │  剩余部分带 Overlap 继续切割
    │
    ▼
第四级：碎片合并（mergeSmallChunks）
    相邻小 chunk 自动合并，不超过 Size+Overlap
```

**Markdown 模式**在此基础上叠加了标题感知层：

```
按标题分段 → 每节独立走上面的流水线 → 跨节携带 Overlap 上下文
                                          ↓
                                    Metadata["heading"] 记录标题来源
```

---

## Q4：`findBestBoundary` 这个函数的设计很有意思，说说你的优先级策略？

**🎯 回答方向**：展示你的细节思考，包括为什么这样排优先级。

边界搜索的目标是：**在尽量靠近预设 Size 的位置，找到最自然的语义断点**。

我设了四级优先级：

| 优先级 | 边界类型 | 举例 | 为什么最高 |
|--------|---------|------|-----------|
| 0（最高） | 段落空行 | `\n\n` | 段落是天然语义单元，在段落间切断几乎无损失 |
| 1 | 句尾 | `。！？.!?` | 句子是完整语义单元 |
| 2 | 换行 | `\n` | 列表项、代码行间的自然分割 |
| 3 | 标点 | `，；,;` | 最后的保底方案，至少不是单词中间切 |

搜索范围限制在 Size 的**后半段**（`Size/2 ~ Size`），避免在 chunk 开头就切断——那样会导致前面一块内容太少。

还有一个细节：**段落边界一旦找到就立即返回**，不继续搜索。因为段落边界是「无损」分割点，不需要再对比其他候选。

## Q5：Markdown 模式下大章节递归切割，标题元数据怎么传递的？

**🎯 回答方向**：展示你对元数据追踪的设计。

当一个大标题下的内容超过 `Size` 时，`splitOversized` 会把内容切成多块。这时候每个子块需要知道「我从哪个标题来的」。

做法是：子块生成后，遍历设置 `chunk.Metadata["heading"] = "核心算法 > 注意力机制"`，并更新 `DocTitle`。

```go
for _, ch := range subChunks {
    if ch.Metadata == nil {
        ch.Metadata = make(map[string]string)
    }
    ch.Metadata["heading"] = sec.headingPath
    ch.DocTitle = fmt.Sprintf("%s > %s", title, sec.headingPath)
}
```

这样下游的检索器可以从 `Metadata["heading"]` 知道这个碎片来自文档的哪个章节，回答时可以带上来源引用。

## Q6：`mergeSmallChunks` 怎么保证合并后不产生超大的块？

**🎯 回答方向**：展示边界约束设计的思考。

合并条件有两个：

```go
// 条件1：当前累计块或新块太小
bufSize < MinSize || curSize < MinSize

// 条件2：合并后不超过 Size + Overlap（允许少量弹性）
bufSize + curSize <= Size + Overlap
```

两个条件**同时满足**才合并。条件 1 确保不会产生碎片，条件 2 确保合并后的 chunk 不会膨胀到远超预设 Size。

`Size + Overlap` 这个上限是刻意设计的——`Overlap` 本身就是「允许的额外上下文」，合并时借用这个余量，不会破坏整体的粒度控制。

## Q7：你怎么验证你的优化是有效的？

**🎯 回答方向**：如果没做正式评估，就说你的验证思路。

项目内置了三篇示例文档：**RAG 技术介绍**、**Go 语言入门**、**VolSeek-Agent 设计文档**。我做了三件事验证：

### 1. 边界质量人工检查

用原来的策略和新策略分别对这三篇文档切片，对比切分点的位置。原来在段落中间切断的地方（比如「RAG 的核心…\n思想是检索+生成」中间切断），新策略会优先在 `\n\n` 处切，保留段落完整。

### 2. chunk 数量分布

旧方案会产生一些 30-50 字的碎片块（短列表项、单行代码示例），新方案的 `mergeSmallChunks` 会把它们合并到相邻块中，chunk 数量减少了约 15-20%，且没有超大型 chunk。

### 3. 检索回归

用三篇文档中的典型问题（"什么是 RAG？"、"goroutine 是什么？"）做检索测试，新策略下 top-3 召回的内容语义更集中，没有出现「半句话+后半段无关内容」的情况。

如果要上生产，下一步我会用**精确率@K**和**召回率@K**做定量评估。

## Q8：你这个方案有什么局限性？还有什么可以继续优化的？

**🎯 回答方向**：主动说出短板，展示你的思考深度。

### 1. 语义分块（Semantic Chunking）

目前还是基于规则（段落、句子边界），没有用 embedding 做语义相似度检测。理想的做法是：先用滑动窗口粗略切分，计算相邻窗口的 embedding 相似度，在相似度骤降的地方切断——这就是「语义边界」。

### 2. 代码块感知

Markdown 中的代码块（`` ``` ``）应该整块保留，不能被中间切断。目前我还没做这个检测。

### 3. 分级标题路径

目前的 `buildHeadingPath` 只返回当前标题文本，没有构建 `h1 > h2 > h3` 这样的完整层级路径。对于复杂的长文档，这会丢失标题间的父子关系。

### 4. 动态 Size 调整

现在的 `Size=500` 是全局固定的。但不同文档类型的最佳块大小不同——技术文档可能需要 800，新闻摘要可能 300 就够了。可以加一个按文档类型自动调整的机制。

## Q9：你提到的「嵌入相似度语义分块」具体怎么做？

**🎯 回答方向**：展示你对前沿方案的了解。

滑动窗口 + cosine similarity 检测：

```
1. 用 100 字符的窗格滑动，步长 50
2. 对每个窗口内容做 embedding
3. 计算相邻窗口的 embedding 余弦相似度
4. 相似度曲线出现「断崖式下跌」的位置 → 语义边界
```

比如一段文本从「RAG 的原理介绍」过渡到「GraphRAG 的架构设计」，在过渡点附近 embedding 相似度会从 0.9 骤降到 0.3，这就是最佳切分点。

这个方案的精度远高于标点符号规则，但代价是多了一次 embedding 调用（有 API 成本）。离线索引时做是值得的，实时检索时可能太慢。
