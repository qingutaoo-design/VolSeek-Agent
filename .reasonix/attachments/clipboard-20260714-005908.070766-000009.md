# 面试问答：RAG 知识库的增量更新策略

> 基于 VolSeek-Agent 项目的文档级哈希对比 + Qdrant 差分更新实现

---

## Q1：你们 RAG 系统的知识库是怎么更新的？每次启动都全量重建吗？

一开始确实是全量重建——启动时清空 Qdrant，重新切片、重新 embedding、重新写入。但这样做有几个问题：

| 问题 | 影响 |
|------|------|
| 启动慢 | 每次都要调 Embedding API，耗时几十秒 |
| 浪费 API 调用 | 明明文档没变，却重新向量化一遍 |
| 不支持增量扩展 | 用户新增一篇文档，必须重建整个库 |

后来改成了**文档级增量更新**：每篇文档分配固定 UUID，计算 SHA256 哈希，启动时逐篇对比，只索引变化的文档。

---

## Q2：具体怎么做增量更新？讲讲设计思路。

核心思路很简单：**给每篇文档一个身份标识，变了才动手**。

### 架构

```
启动 → 遍历文档列表
        │
        ├── 计算 SHA256(content)
        ├── Qdrant 中查 doc_uuid
        │     ├── 哈希一致 → 跳过（⏭️）
        │     └── 哈希不同/不存在 → 删除旧 chunk → 重新切片/embedding → 写入
        └── 结束
```

### 关键数据结构

每篇文档存成一个 `docEntry`：

```go
type docEntry struct {
    uuid    string  // 固定 UUID（doc-rag-intro, doc-go-intro...）
    title   string  // 显示标题
    content string  // 文档内容
}
```

### Qdrant 中的 payload 设计

```go
// 每个 chunk 写入 Qdrant 时带以下 payload：
{
    "content":   chunk 原文,
    "doc_title": "RAG 技术介绍",
    "doc_uuid":  "doc-rag-intro",   ← 文档级标识
    "doc_hash":  "sha256 哈希值",    ← 用于对比
    "index":     0                   ← chunk 序号
}
```

### 增量更新流程

```go
for _, doc := range docs {
    hash := sha256.Sum256([]byte(doc.content))
    hashStr := hex.EncodeToString(hash[:])

    // 步骤 1：查 Qdrant 中是否有相同哈希
    if checkDocHash(qs, doc.uuid, hashStr) {
        skippedCount++  // 哈希一致 → 跳过
        continue
    }

    // 步骤 2：删除该文档所有旧 chunk
    qs.DeleteByDocUUID(context.Background(), doc.uuid)

    // 步骤 3：重新切片、embedding
    chunks := chunker.Chunk(doc.content, doc.title)
    allDocs = append(allDocs, chunks...)
    changedCount++
}

// 步骤 4：批量写入 Qdrant（带 doc_uuid 和 doc_hash）
qs.AddBatchWithMeta(docChunks, doc.uuid, hashStr, true)
```

## Q3：Qdrant 怎么支持按文档 ID 删除的？你的 `DeleteByDocUUID` 怎么实现的？

Qdrant 支持用 payload 字段做过滤删除。`DeleteByDocUUID` 本质上就是发一个带 filter 的 Delete 请求：

```go
func (qs *QdrantStore) DeleteByDocUUID(ctx context.Context, docUUID string) error {
    _, err := qs.client.Delete(ctx, &qdrant.DeletePoints{
        CollectionName: qs.collection,
        Wait:           qdrant.PtrOf(true),
        Points: qdrant.NewPointsSelectorFilter(&qdrant.Filter{
            Must: []*qdrant.Condition{
                qdrant.NewMatch("doc_uuid", docUUID),  // 只删这 UUID
            },
        }),
    })
    return err
}
```

这比全量删重建高效得多——只修改一篇文档的 chunk，其他文档完全不受影响。

## Q4：新增文档怎么办？比如用户往项目里加了第 4 篇文档。

加一篇新文档只需要在 `docs` 列表里加一个 entry：

```go
docs := []docEntry{
    {"doc-rag-intro", "RAG 技术介绍", `...`},
    {"doc-go-intro", "Go 语言入门", `...`},
    {"doc-volseek", "VolSeek-Agent 设计文档", `...`},
    {"doc-new", "新文档标题", `新文档内容...`},  // 新增
}
```

启动时：
1. `doc-new` 在 Qdrant 中不存在 → `checkDocHash` 返回 false
2. `DeleteByDocUUID` 不会删任何东西（没有要删的）
3. 正常切片、embedding、写入

已有 3 篇文档的哈希没变，毫秒级跳过。

---

## Q5：你怎么保证 `checkDocHash` 的效率？会不会每次启动都扫全库？

不会。`checkDocHash` 用的是 Qdrant 的 `Scroll` 接口，**带 filter 只查 1 条**：

```go
func checkDocHash(qs *rag.QdrantStore, docUUID, expectedHash string) bool {
    points, err := qs.ScrollByFilter(context.Background(),
        map[string]string{"doc_uuid": docUUID},  // 过滤条件
        1,                                        // 只查 1 条
    )
    if err != nil || len(points) == 0 {
        return false
    }
    if v, ok := points[0].Payload["doc_hash"]; ok {
        return v.GetStringValue() == expectedHash
    }
    return false
}
```

每次调用只查 1 个 point，O(1) 时间。3 篇文档总共 3 次 Scroll 请求，毫秒级完成。即使扩展到 1000 篇文档也只是 1000 次独立小查询。

如果要进一步优化，可以用 count API 先判断 collection 是否为空，或者用 scrollAll 一次拿到所有 doc_hash。

## Q6：内存模式下还能用吗？这个增量更新只对 Qdrant 有效？

对，目前只对 Qdrant 有效。内存模式 (`VectorStore`) 下仍然全量重建——因为内存模式本身没有持久化，每次启动都是空的，不存在"跳过"的概念。

在代码里，这个判断只对 Qdrant 生效：

```go
if qs, ok := store.(*rag.QdrantStore); ok {
    if checkDocHash(qs, doc.uuid, hashStr) {
        needsIndex = false  // Qdrant 模式：跳过
    }
}
```

如果 store 是 `rag.VectorStore`（内存模式），`store.(*rag.QdrantStore)` 类型断言失败，`ok` 为 false，走全量索引路径。

---

## Q7：如果用户改了文档内容但 UUID 没变，会发生什么？

改内容 → SHA256 哈希变化 → `checkDocHash` 返回 false → 进入索引流程：

1. `DeleteByDocUUID` 删除该 UUID 的所有旧 chunk
2. 重新切片（新内容）
3. 重新 embedding
4. 写入新 chunk（带新 hash）

旧文档的 chunk 完全被替换，Qdrant 中不会留下旧数据的痕迹。

---

## Q8：如果 Qdrant 挂了，系统会怎么样？

`cmd/main.go` 里有一个降级逻辑：

```go
qs, err := rag.NewQdrantStore(...)
if err != nil {
    log.Printf("Warning: Qdrant init failed (%v), falling back to in-memory store", err)
    store = rag.NewVectorStore()  // 降级到内存
}
```

Qdrant 连不上 → 自动降级到内存 VectorStore。代价是每次启动全量重建，但系统不崩溃。

---

## Q9：这个方案还有什么可以优化的？

几个方向：

1. **异步 hash 检查** — 目前串行逐个检查文档哈希，如果知识库有上千篇文档，可以并发检查，用 `sync.WaitGroup` 或 errgroup 并行化。

2. **内存模式也支持增量** — 给 VectorStore 加一个本地文件持久化，启动时对比文件修改时间戳，避免全量重建。

3. **批量 upsert 异步化** — 当前 `AddBatchWithMeta` 的 `wait=true` 是同步等写入完成。对于大批量写入，可以设为 false 异步写入，速度更快（但宕机可能丢数据）。

4. **doc_hash 索引** — Qdrant 支持在 payload 字段上创建标量索引。对 `doc_uuid` 建索引后，ScrollByFilter 的查询更快。

---

## 面试加分小结

| 能力点 | 体现在哪里 |
|--------|-----------|
| **增量思维** | 不做全量重建，只索引变化的部分 |
| **存储设计** | payload 中带 doc_uuid + doc_hash，为差分更新打基础 |
| **降级容错** | Qdrant 不可用时自动降级到内存，系统不崩 |
| **效率意识** | checkDocHash 只查 1 条，O(1) 不扫全库 |
| **可扩展性** | 新增文档只需加一个 entry，不影响已有数据 |
