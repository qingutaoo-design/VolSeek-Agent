# 向量存储演进：从内存暴力搜索到 Qdrant 专业向量数据库

> VolSeek-Agent 项目向量存储的技术选型与集成全记录

---

## 一、起点：内存 VectorStore 的局限

### 1.1 原始实现

项目最初使用一个内存向量存储（`internal/rag/rag.go:452-460`），核心结构非常简单：

```go
type VectorStore struct {
    mu       sync.RWMutex
    chunks   []*types.Chunk
    docIndex map[string][]*types.Chunk
}
```

### 1.2 三个致命局限

| 局限 | 说明 | 影响 |
|------|------|------|
| **无持久化** | 所有数据在内存中，程序重启即丢失 | 每次启动都要重新索引文档、重新 embedding、重新构建 |
| **暴力搜索 O(n)** | `Search()` 遍历所有 chunk 算 cosine similarity | 10 万条时单次搜索 ~200ms，百万级不可用 |
| **单机内存上限** | 10 万条 × 1536 维 × 4 字节 ≈ 6GB | 无法扩展到更大规模 |

代码注释自己也承认了这一点（第 454 行）：

```go
// 生产环境建议替换为 pgvector / Qdrant 等专业向量数据库。
```

## 二、为什么需要一个专业向量数据库

### 2.1 业务需求驱动

| 需求 | 内存方案 | 向量数据库方案 |
|------|---------|--------------|
| 重启后数据不丢 | ❌ 每次重新索引 | ✅ 持久化到磁盘 |
| 增量添加文档 | ❌ 只能批量重建 | ✅ 支持 upsert |
| 快速搜索（万级以上） | ❌ O(n) 暴力搜索 | ✅ HNSW 索引 O(log n) |
| 元数据过滤 | ⚠️ 手动实现 | ✅ 原生支持 |
| 备份恢复 | ❌ 无 | ✅ 快照/WAL |

### 2.2 当前规模评估

虽然当前只有 ~42 个 chunk（3 篇文档），但 RAG 系统的知识库增长很快：

- 10 篇文档 → ~150 个 chunk
- 100 篇文档 → ~1500 个 chunk（内存方案开始吃力）
- 10000 篇文档 → ~150000 个 chunk（内存方案不可行）

## 三、候选方案对比

### 3.1 评估维度

| 维度 | 权重 | 说明 |
|------|------|------|
| **Docker 部署复杂度** | ⭐⭐⭐⭐⭐ | Go 项目，需要开发者快速上手 |
| **Go 客户端成熟度** | ⭐⭐⭐⭐⭐ | 必须官方维护、活跃更新 |
| **索引性能** | ⭐⭐⭐⭐ | HNSW 等 ANN 算法 |
| **资源占用** | ⭐⭐⭐ | 镜像大小、内存消耗 |
| **持久化能力** | ⭐⭐⭐ | 备份、恢复、WAL |
| **过滤能力** | ⭐⭐⭐ | 元数据条件过滤 |

### 3.2 最终对比

| 特性 | **Qdrant** | Milvus | Weaviate | Chroma | pgvector |
|------|-----------|--------|----------|--------|----------|
| Docker 部署 | ✅ 单容器 | ❌ 多容器 | ✅ 单容器 | ✅ 单容器 | ✅ 单容器（需 PG） |
| Go 客户端 | ✅ 官方维护 | ⚠️ 已归档 | ⚠️ 61 stars | ❌ 无 | ✅ 345 stars |
| 索引 | HNSW（Rust） | IVF/HNSW | HNSW（Go） | HNSW | HNSW/IVFFlat |
| 镜像大小 | **~180 MB** | ~700 MB | ~1 GB | ~500 MB | ~400 MB |

### 3.3 部分候选的排除理由

**Chroma** — 直接排除。无官方 Go 客户端，只有 Python/JS/Rust，Go 项目无法接入。

**Milvus** — 排除。即使 standalone 模式也需要 etcd + MinIO 两个额外容器。Go SDK 旧版已归档（2025 年 3 月），新版嵌入主库不稳定。

**Weaviate** — 排除。Go 客户端仅 61 stars，社区活跃度远低于服务端（16.6K stars）。容器镜像约 1GB。

**pgvector** — 备选。如果项目已使用 PostgreSQL，pgvector 是最优选择，SQL 过滤能力最强。但本项目是轻量级 Agent，不希望引入关系型数据库。

## 四、为什么选 Qdrant

### 4.1 六大理由

#### 理由 1：部署极简

```bash
docker run -d --name qdrant -p 6333:6333 -p 6334:6334 qdrant/qdrant
```

单容器、零依赖、镜像仅 180MB。

#### 理由 2：Go 客户端成熟可靠

`github.com/qdrant/go-client/qdrant`（135+ importers）：
- 基于 Protobuf 自动生成，与服务端版本严格匹配
- 完整的 gRPC 接口：upsert/query/delete/scroll/filter
- 类型安全，编译期发现接口不匹配

```go
// 接入代码示意
client, _ := qdrant.NewClient(&qdrant.Config{Host: "localhost", Port: 6334})
client.Upsert(ctx, &qdrant.UpsertPoints{
    CollectionName: "volseek",
    Points:         []*qdrant.PointStruct{...},
})
```

#### 理由 3：Rust 原生 HNSW

Qdrant 的 HNSW 实现用 Rust 编写，性能优于 Weaviate 的 Go 实现。当前规模（42 个 chunk）差异不大，但扩展到百万级时 Rust HNSW 的优势明显。

#### 理由 4：内存效率

| 规模 | Qdrant 预期内存 | 内存 VectorStore |
|------|----------------|-----------------|
| 100 条 | ~50 MB | ~5 MB |
| 1 万条 | ~150 MB | ~600 MB |
| 10 万条 | ~500 MB | ~6 GB（不可行） |

#### 理由 5：丰富的过滤能力

```go
// 支持 must/should/must_not 组合、范围、文本匹配
filter := &qdrant.Filter{
    Should: []*qdrant.Condition{
        qdrant.NewMatch("doc_title", "RAG 技术介绍"),
    },
}
```

#### 理由 6：持久化

数据写入磁盘（`/qdrant/storage`），支持 WAL 和快照。容器重启数据自动恢复。

## 五、架构变更

### 5.1 接入前

```
indexSampleDocuments()
    → chunker.Chunk()
    → llmClient.EmbedBatch()
    → vectorStore.AddBatch()     ← 内存存储
    → graphStore.BuildFromChunks()
```

### 5.2 接入后

```
indexSampleDocuments()
    → chunker.Chunk()
    → llmClient.EmbedBatch()
    → store.AddBatch()           ← Store 接口，实际为 QdrantStore 或 VectorStore
    → graphStore.BuildFromChunks()

         ┌──────────────────────────────────────────────┐
         │  Store 接口                                   │
         │  ├── Add(chunk)                              │
         │  ├── AddBatch(chunks)                        │
         │  ├── Search(queryEmbed, topK, threshold)     │
         │  ├── GetAllChunks()                          │
         │  └── Len()                                   │
         ├────────────────┬─────────────────────────────┤
         │ VectorStore     │ QdrantStore                 │
         │ (内存，原方案)   │ (Qdrant gRPC 客户端)        │
         └────────────────┴─────────────────────────────┘
```

### 5.3 关键设计决策

1. **定义 `Store` 接口**：两个存储实现都遵循同一接口，`Retriever` 通过接口调用，无需关心底层实现
2. **按环境变量切换**：`QDRANT_HOST` 有值用 Qdrant，否则用内存——开发环境零配置
3. **维度可配**：`QDRANT_DIMENSION`（默认 1024），适配不同 embedding 模型
4. **QdrantStore 保留本地 chunk 缓存**：用于 `keywordSearch` 和 `Len()`，避免每次查询都访问 Qdrant

## 七、未来优化方向

1. **自动维度检测**：从第一个 chunk 的 embedding 自动推断维度，避免手动配置
2. **批量 upsert 优化**：当前每批 100 条，可根据文档量动态调整
3. **collection 版本管理**：支持多版本 collection，方便 embedding 模型升级时迁移
4. **连接池与重试**：生产环境需要 gRPC 连接池和断线重试机制
